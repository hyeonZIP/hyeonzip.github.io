### 2. OOP 
지난 코드 리뷰에서 받은 피드백을 바탕으로 클래스를 세분화 하였다.
+ Controller
    + RacingController
    > Controller는 View와 Service사이에서 중계 역할을 한다.
+ Service
    + RacingService
    > Model과 연관되며 Racingcar 객체를 생성하고 Racingcars 객체 리스트를 생성한다.
+ View
    + InputView
    > 사용자 입력에 관련된 함수가 정의되어있다.
    + OutputView
    > 출력과 관련된 함수가 정의되어있다.
+ Model
    + Racingcar
    > 자동차 이름과 이동 거리가 저장되는 객체이다.
    + Racingcars
    > Racingcar객체를 객체 리스트로 저장한다.
+ utils
    + StringReplacer
    > 이 부분은 모험적으로 분리를 해보았다. 문자열 사이의 공백 제거 기능을 제공한다.
    + StringSplitter
    > 콤마 기준으로 입력된 문자열을 분리하는 기능을 제공한다.
+ validate
    + RacingcarAttemptValidate
    > 시도 횟수 입력 과정의 검증 기능을 제공한다.
    + RacingcarNameValidate
    > 자동차 이름 입력 과정의 검증 기능을 제공한다.

## 🤓Learned
이번 2주차에서 가장 고민을 한 부분은 총 3가지로
1. Racingcar 객체의 distance 자료형을 int로 할지 String으로 할지
2. 차수별 전진 현황을 나타낼 때 Random()값에 의한 전진 로직을 어디에 둘지
3. Controller에서 Service로 옮길 수 있는 기능이 뭐가 있을까?

1번은 단순히 매 차수별 실행 결과를 나타낼 때 `int만큼 '-'를 출력하는 것` 보다

`String으로 받고 그대로 출력하는 것`이 로직이 더 간단할 것이라고 생각했다.
<hr>
2번은 객체의 `정보 은닉`을 위한 getter/setter 지양을 위해 사용해보지 않고 구현하기로 했다.

그럼 어떻게 구현해야 하는가?

완성된 결과를 리턴하는 방식으로 정보 은닉을 하기로 했다.

Racingcars객체 리스트에서 각 객체를 순회하며 Racingcar 객체 안에 Random()로직을 구현하였다.

만약에 시청자 투표로 인해 승리자가 정해지는 로직이였다면 투표 로직을 Racingcar와 별도로 구현했겠지만

Random()을 통해 전진하는 것이 결국 자동차의 기능 일부라고 생각하여 Racingcar 객체 안에 구현했다.
<hr>
마지막으로 3번은 아직까지도 찝찝하다.

교내 프로젝트에서 Spring프로젝트를 할 때의 Controller는 하나의 기능에 대한 로직을 구현하다 보니

Controller의 코드줄은 비교적 짧고 Service에 비즈니스로직을 구현하다보니 코드줄이 길었다.

하지만 현재의 방식은 Controller에 필요한 field를 선언하고 프로그램의 시작부터 끝까지의 과정이 모두 담겨있다보니

이전까지의 과정이 오버랩되면서 이제까지 해온 방식이 잘못된건지 혼란스러웠다.

따라서 이 부분은 이번 코드 리뷰를 통해 새롭게 받아보기로 한다.
<hr>
이번에 여러 클래스로 분리하는 것이 테스트 케이스를 작성하는데 유리하다는 말의 의미를 확실하게 알았다.

테스트 케이스를 작성할 때 필요한 메소드만 호출하면 되니 쉽고 빠르게 테스트를 진행할 수 있었다.

기존에는 검증 로직이나 연산 로직이 예상한 대로 동작하는지 확인하기 위해 각 줄마다 sout을 입력하고

Run > 코드 수정 > Run > 코드 수정 > Run > 코드 수정 > Run 의 반복이였다.

하지만 테스트 코드를 작성해두니 내가 이전에 검증한 동작 방식을 확인할 수 있고 본 코드에는 깔끔하게 유지할 수 있어서 보기 편했다.
